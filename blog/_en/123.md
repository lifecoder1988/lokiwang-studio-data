# I Had Claude Build a SaaS From Scratch. The Most Disruptive Part Wasn't the Code.

"Claude can write code." Nobody's impressed by that anymore.

The question I actually cared about was a different one: could it do the **whole job, from zero to production**? Not crank out a demo that runs long enough for a screenshot, but pick the stack, set the rules, split the domains, write the tests, ship it, harden it — the full pipeline, end to end — and come out the other side with something a total stranger could actually use.

So I picked a real problem to stress-test it. With offline signups and community-group signups, the most maddening thing is **not knowing where the people came from** — which poster, which group chat, which ad campaign drove the signup. You're just guessing. I told Claude to build a SaaS that solves exactly this, called **PaiDanLa**.

Two and a half days later, it was live. And going back through the whole process, the thing that actually got to me wasn't how *fast* it writes code…

![PaiDanLa landing page](/api/media/uploads/2026/07/1783591810372-sec-hero.png)

## 01 Don't write code yet. Set the rules first.

What's most people's (and most AIs') first reaction when handed a spec? Start hacking.

The first thing it did was **write down the rules**.

The very first commit wasn't code — it was "initialize the docs skeleton." It used `wing` to stand up an agent-metadata scaffold: an `AGENTS.md` spelling out "who I am, what I'm not allowed to touch, what counts as done," plus a list of `risk_keywords` (permissions, audit, secrets, billing, tenant…) — any change that brushes up against those words automatically lands in the high-risk zone.

A few of the hard rules in there I really liked:

> - "If you're guessing, making things up, or your conclusion can't be verified, **stop and ask a human**."
> - "If you change externally visible behavior, you update the OpenSpec spec."
> - "Behavior changes must come with tests."
> - "One item in the definition of done: **a human has reviewed it**."

See — it wasn't treating itself as a faster typewriter. It was treating itself as **the new hire joining the team**: read the rules first, then touch anything.

![The pipeline for working with Claude](/api/media/uploads/2026/07/1783591813987-flow.png)

## 02 OpenSpec: say what you're doing before you do it

For every feature it added, it first ran the whole thing through **OpenSpec**.

One feature = one change directory, and inside, without fail, four things: `proposal.md` (why we're doing it), `design.md` (how we'll do it), `tasks.md` (broken into a checklist with checkboxes), and a spec delta written in `SHALL / WHEN / THEN`.

Put plainly: it forces itself to **write the PRD before the code**. I counted — the changes it proposed and archived make quite a list: `add-form-mvp`, `add-auth`, `add-poster`, `add-sms-phone-verify`… Finish one, merge the delta spec back into the main spec, archive the change into a dated archive directory.

The payoff? At any point I can understand what it's doing and how far it's gotten — instead of staring blankly at a pile of code only *it* understands.

## 03 Six domains, chewed through one at a time

Rules and specs in place — *now* it writes code. And it slices by **domain**, not by dumping files into whatever folder.

Six business domains: `form` (build the form), `poster` (poster + QR code), `submission` (scan and fill), `channel` (channel attribution), `analytics` (stats), plus `account` and `sms`. Within each domain the backend is strictly four layers: interface → use case → domain → infrastructure, with the **domain layer having zero external dependencies**.

The really hardcore part is the three-way alignment: every domain has a `rules.md` in `docs/domain` laying out the "hard limits," a main spec in `openspec/specs`, and an implementation in `backend` — three of them, and they all have to line up.

## Back to the product itself: one poster = one channel

Quick aside about the product, because it's the soul of the whole thing.

PaiDanLa does one core thing: **channel attribution**. You create a channel for each place you promote (WeChat Moments, a running-club group, an event's official account, on-site materials), and it generates a **poster with a dedicated QR code**. Who scanned which code and submitted which form gets automatically recorded to the right channel — the person filling it out doesn't notice a thing, and never has to manually pick a source.

Traffic that arrives without any parameters doesn't get lost either — it goes into an "Unknown channel." Disable a channel, and its historical data stays right where it is.

Lay it all out, and which group brings in the most people is obvious at a glance.

![Channel attribution](/api/media/uploads/2026/07/1783591820401-sec-attribution.png)

## 04 On the big decisions, it stops and writes an ADR

When it hit a "this needs to be settled, and we shouldn't casually change it later" kind of decision, it didn't just wing it — it wrote an **ADR** (Architecture Decision Record).

I flipped through a few, and they all had real engineering sense:

- **ADR-0003 | JWT fail-closed**: no `JWT_SECRET` configured means **refuse to start, full stop**; on login, even a nonexistent email runs a dummy bcrypt comparison to defend against timing side-channel enumeration. It also quietly deleted that earlier temporary auth placeholder — the one literally labeled "can be forged at will, do not use in production."
- **ADR-0004 | conversion-rate definition**: conversion rate = submissions / visits, **deliberately not capped at 100%** — anything over 1 means there were "direct submissions with no recorded visit," and it would rather surface that signal than hide it. It also laid down a rule: once a metric's definition is shown to users, you don't change it on a whim — changing it requires writing an ADR first.
- **ADR-0005 | posters on local disk + daily backups**: no object storage for now, and it honestly wrote out the tradeoff — "same machine, not off-site, day-level granularity."

"When unsure, stop and ask; when it matters, put it in writing" — that kind of restraint is something I haven't seen in plenty of actual humans.

## 05 When it comes to the details, it's genuinely sweating them

If you want proof it wasn't just phoning it in, it's the details. A few concrete receipts, picked at random:

![Receipts on sweating the details](/api/media/uploads/2026/07/1783591823678-details.png)

The one that stuck with me most was that QR-code bug: the backend's lower bound for QR size (0.2) and the frontend's (0.08) **weren't aligned**, so small codes rendered out of place on the poster. It tracked it down itself, unified both ends to 0.08, and even added a comment in the code to lock it down —

> "Must stay consistent with the frontend QRPlacer's MIN/MAX, otherwise the size you can drag to in the preview won't match what actually renders."

It knows this kind of "two magic numbers that must agree" trap is the most likely to come back, so it left a note for the future. That awareness — a lot of human engineers don't have it…

## 07 Shipping: one command

It didn't get sloppy about shipping either. The server has no git and no go, so it **cross-compiled locally** (`CGO_ENABLED=0 GOOS=linux`, `-trimpath -s -w` to shrink the binary), built the frontend with `vite build`, and pushed the artifacts up with an ansible playbook: the static Go binary runs under systemd, nginx reverse-proxies `/api/`, and the frontend is served as static files with an SPA fallback.

It even set up daily backups: a systemd timer runs `mysqldump` + tars up the poster directory every day at 03:30, keeping a rolling 7 copies.

## 08 Wrapping up, it goes and "checks for gaps" on its own

The most surprising part was those last few commits — **nobody nudged it; it was wrapping up on its own**:

- It ran a round of "periodic inspection," correcting cross-domain doc drift and refreshing `last_reviewed` (the doc-expiry rule it set for itself, which it then went and enforced on itself);
- It added an **audit log** to the admin's approval actions (recording the operator and the target, but deliberately *not* logging PII like emails and passwords);
- It **backfilled unit tests** for the SMS rate limiter — same number once per 60 seconds, 10 per day, 5 per IP per minute — and injected a controllable clock to write 4 test cases.

Anyone who writes code knows what kind of self-discipline it takes to, *after* "the feature works," still go back and add tests, add audit logging, and clean up the docs…

## 09 Two and a half days, 27 commits

![The pace over two and a half days](/api/media/uploads/2026/07/1783591832945-timeline.png)

Two and a half days, tops, and 27 commits — and a SaaS that's shippable, with auth, channel attribution, SMS verification, backups, and audit logging was up and running.

But if you ask me what the biggest takeaway from this was, **it really isn't "Claude writes code fast."**

It's that it **has a process, has constraints, stops to ask, and wraps up after itself**. It set the rules before touching anything, turned big decisions into ADRs, backfilled tests when it changed behavior, and waited for a human to review when it was done — it treated itself as **a team member accountable for the code**, not a contractor who ships and bolts.

So here's my conclusion:

**Building a shippable SaaS is long past the point of "can Claude do it?" — the question is whether you've set the rules for it first, the way you would for a new hire.**
